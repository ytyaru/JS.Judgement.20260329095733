# judgement.js 合否判定API

## 背景

プログラミングは単一責任の原則に従い、特定の入出力をする関数が最小単位である。この時、出力は成否（成功／失敗）の二択に大別できるはずだ。しかし、現行の標準APIでは、成否を統一した記述で判定する方法がない。そこで、成否判定する統一APIとしてjudgement.jsを実装することを企画した。

### 現行の問題点

例えばArray.prototypeの`indexOf`と`find`という標準APIがある。これらは失敗時にそれぞれ`-1`を返したり`undefined`を返したりする。つまりAPIによって失敗時の返却値が不定だ。そのせいで、条件式を変えねばならなかったり、その値が失敗を意味することだと見て分かりにくかったりする問題がある。

```
const index = str.indexOf('abc', 'z');
if (-1 === index) {return '失敗'}
```
```
const item = [0,1,2].find(v=>9<v);
if (undefined === item) {return '失敗'}
```

### judgement.jsによる改善案

成否は`isPass`/`isFail`ゲッターにより真偽値を返すことで判定できるようにする。これなら失敗時APIごとに違う値を気にすることなく、常に`isPass`/`isFail`という読んで字のごとく意図を理解できる英語で結果を読み取れる。

問題は結果の受取が二段構えになってしまうことだ。`res`という一時変数が新たに必要になってしまう。一旦`res`で受け取り、失敗ルートを排除してから、ようやく`index`という成功時の結果を受け取る必要が産まれてしまう。二度手間だ。

```
const res = str.indexOf('abc', 'z');
if (res.isFail) {return '失敗'}
const index = res.at();
```

`res`を受け取らず、失敗したら例外発生して中断することもできる。この場合は`res`は不要だが、成否ごとにルート分岐することができなくなる。失敗したら例外発生して強制的に中断されてしまう。

```
const index = str.indexOf('abc', 'z').throw();
```

例外発生した時の制御はJS標準構文の`try catch`で可能だ。例えば以下のように。

```
try {
    const index = str.indexOf('abc', 'z').throw();
} catch (e) {
    throw new MyAppError(`アプリケーションエラーです。`, {cause:e});
}
```

しかしJudgement実装したものはアプリ側からみれば利用したAPIだ。責任分離したエラーメッセージにしたい場合は次のように条件分岐することになる。

```
try {
    const index = str.indexOf('abc', 'z').throw();
} catch (e) {
    if (e instanceof JudgementFailure) {
        throw new MyAppError(`APIエラーです。`, {cause:e});
    } else {
        throw new MyAppError(`アプリケーションにとって予期せぬエラーです。`, {cause:e});
    }
}
```

しかし、`try catch`に`if`とネストが深くなってしまう。また、例外の型で判定せねばならないのも面倒だ。この場合まずはその型をjudgement.jsからインポートする所から始めねばならないのも不満点の一つだ。

そもそも、`try catch`のほうを大枠にすることに違和感がある。エラー時の挙動は子要素として内部に押し込めるほうが自然に思える。あるいは同列階層において、次の行に書くという同列上の順序として書くのが自然に見えるはず。どうにも`try catch`という構造が気に食わない。

そこで以下のようなコールバック関数による解決を考案した。

```
str.indexOf('abc', 'z')
.pass(index=>{
    alert('成功!!: ', index);
})
.fail(res=>{
    document.body.appendChild(res.cause.constructor.name);
    document.body.appendChild(res.cause.message);
    throw new MyAppError(`APIエラーです。`, {cause:res.cause});
})
.throw();
```

しかしこれはアプリケーションエラーとの分配ができていない。それを実装すると以下だ。しかし以下はネストの深さが再発してしまう。その上、`try catch`が`pass`と`fail`で重複コードになってしまいDRYに書けないという新たな問題が発生する。

```
str.indexOf('abc', 'z')
.pass(index=>{
    try {
        alert('成功!!: ', index);
    } catch (e) {
        throw new MyAppError(`アプリケーションにとって予期せぬエラーです。`, {cause:e});
    }
})
.fail(res=>{
    try {
        document.body.appendChild(res.cause.constructor.name);
        document.body.appendChild(res.cause.message);
        throw new MyAppError(`APIエラーです。`, {cause:res.cause});
    } catch (e) {
        throw new MyAppError(`アプリケーションにとって予期せぬエラーです。`, {cause:e});
    }

})
.throw();
```

これらを解決するために`.catch()`を考案した。すなわち`pass()`や`fail()`で例外発生した時は、`catch()`に渡されたコールバック関数で処理をするという構造だ。

```
str.indexOf('abc', 'z')
.pass(index=>{
    alert('成功!!: ', index);
})
.fail(res=>{
    document.body.appendChild(res.cause.constructor.name);
    document.body.appendChild(res.cause.message);
    // res.throw()を使うことで`.catch()`が動作することを回避する（想定通りの例外発生である）
    res.throw(MyAppError(`APIエラーです。`, {cause:res.cause}));
//    throw new MyAppError(`APIエラーです。`, {cause:res.cause});
})
.catch(e=>{// 想定外の例外発生である場合（pass()/fail()内の実装で想定外の例外(res.throw()以外で発生した例外)が発生した場合）
    throw new MyAppError(`アプリケーションにとって予期せぬエラーです。`, {cause:e});
})
.throw();
```

このコードには以下三点の問題がある。

1. `catch()`時は発生箇所の特定が必要(`pass()`/`fail()`)
2. `fail()`時に「予期した例外」と「予期せぬ例外」の区別が必要
3. コールバック関数のネスト地獄が発生しうる

一つずつ見ていこう。

まず、1.`catch()`時は発生箇所の特定が必要(`pass()`/`fail()`)について。すぐ思いつくのはコールバック関数の引数を追加することだ。以下のように`isPass`を追加する。

```
.catch((e,isPass)=>{
    throw new MyAppError(`アプリケーションにとって予期せぬエラーです。これは${isPass ? 'pass()' : 'fail()'}で発生しました。`, {cause:e})
})
```

しかしそれだと冗長に思える。では、`catch()`のコールバック関数の引数を`res`(`JudgementResult`(`Pass`/`Fail`))にすると良いか？　だがPassには`cause`プロパティが存在しない。いや、そもそもここではPassが返るはずはないか。例外発生したから`catch()`しているのだから。となると`Fail`になるのか？　だが`Fail`は実装された予期した失敗であるのに対し、`catch()`は呼出側の実装内で予期せぬ例外が発生した時の話だ。よって`Fail`にするのもおかしい。となると、ここで渡されるべきは、まだ存在しない新しい概念になるべきだ。すなわち「呼出側にとって想定外のエラーである」ことを意味するものだ。

```
class JudgementCalledUnexpectedError extends JudgementError {
    constructor(msg, options) {
        super(msg, options);
        this.name = 'JudgementCalledUnexpectedError';
        if ('Object'!==getId(options)) {throw new ImplementationError(`optionsは必須です。{isPass:false}等`)}
        if (!options.hasOwnProperty('isPass')) {throw new ImplementationError(`optionsにisPassキーは必須です。{isPass:false}等`)}
        if ('boolean'!==typeof options.isPass) {throw new ImplementationError(`options.isPassはtrueかfalseであるべきです。`)}
        this._isPass = options.isPass;
    }
    get isPass() {return this._isPass}
}
```
```
.catch(e=>{
    throw new MyAppError(`アプリケーションにとって予期せぬエラーです。これは${e.isPass ? 'pass()' : 'fail()'}で発生しました。`, {cause:e})
})
```

二つ目の課題。2. `fail()`時に「予期した例外」と「予期せぬ例外」の区別が必要について。

`catch()`を新設したことで、`fail()`時に、予期した例外と、予期しない例外の区別をつける必要があるという新たな課題が発生する。それは利用者にとって理解しづらい概念だろう。素直に`throw`構文を使いたくなるに違いない。だがそれだとその例外を内部で噛み殺して`.catch()`が実行されてしまうため期待通りの動作にはならない。さもなくば`catch()`が機能しなくなってしまうからだ。

`fail(res=>{})`のコールバック関数内における例外発生については特殊なAPIによってしか意図した例外発生ができない、というオレオレルールができてしまう。

これについてはもう、`catch()`を実装する以上は覚えてもらうしない。もしそれが嫌なら`catch()`を廃止し、`pass()`と`fail()`にそれぞれ重複する`try catch`コードを書くというDRYに書けない保守性の低いコードを容認するか、あるいはそれを`try catch`の中で書くという読みづらいコードを容認することになる。

オレオレルールと`try catch`親配置、どっちがマシか？　Judgementを実装する以上、オレオレルールを推すしかない訳だが。それが初見の人にとって理解しづらいことも理解できてしまうので売り込みづらい。

三つ目の課題。3. コールバック関数のネスト地獄が発生しうる。について。

残念ながら、コールバック関数におけるネスト地獄が発生しうる。以下のように。こうなるとネストが深くなりすぎて読みにくい。

```
str.indexOf('abc', 'z')
.pass(index=>{
    // Judgementを返す別のAPI
    someJudgementApi1(index);
    .pass(res=>{
        // Judgementを返す別のAPI
        someJudgementApi2(res);
        .pass(res=>{
            // 以下JudgementAPIを使った数だけネストが深まっていく……
        })
        .fail(res=>{})
        .catch(e=>{})
        .throw();
    })
    .fail(res=>{})
    .catch(e=>{})
    .throw();
})
.fail(res=>{
    document.body.appendChild(res.cause.constructor.name);
    document.body.appendChild(res.cause.message);
    res.throw(MyAppError(`APIエラーです。`, {cause:res.cause}));
})
.catch(e=>{
    throw new MyAppError(`アプリケーションにとって予期せぬエラーです。`, {cause:e});
})
.throw();
```

これは致命的な自己矛盾が存在する。

ネストを深めぬようにコールバック関数にしたのに、そのコールバック関数がネストする構造を作り込んでしまっているという矛盾が発生している。

なので結局、最初の戻り値を返す方式に回帰する。

```
const res = str.indexOf('abc', 'z');
if (res.isFail) {return '失敗'}
const index = res.at();

const res1 = someJudgementApi1(index);
if (res1.isFail) {return '失敗'}
const ans1 = res1.at();

const res2 = someJudgementApi1(ans1);
if (res2.isFail) {return '失敗'}
const ans2 = res2.at();
```

つまり、Judgementを利用したAPIを一つしか使っていないなら、コールバック関数方式のほうが可読性が高いが、Judgementを利用したAPIを複数使うなら、ネスト地獄を避けるために二段構え方式で書いたほうが可読性が高い。

このことから、Judgementは両方の記述を可能にする必要がある。そしてJudgement利用者は、こうした実装方法があることを把握し、今、どの記法を選ぶべきか決定せねばならないという負荷を負うことになる。

しかもその内部で使うコールバック関数の引数には、オレオレルールを含んだ特殊な記述が必要になる。

果たしてその負荷は、統一APIで記述可能になった利点と比較して、効果的であり有意義であり価値があると言えるだろうか？　以下のようにAPIごとに違う失敗値をリテラルで書いて分岐するという覚えゲーと、Judgementのオレオレルールを覚えるのと、どちらが大変だろう？

```
const index = str.indexOf('abc', 'z');
if (-1 === index) {return '失敗'}
```
```
const item = [0,1,2].find(v=>9<v);
if (undefined === item) {return '失敗'}
```

他の懸念点がある。メソッドチェーン時、`throw()`が最後なのは順序がおかしくないか？　実行順と表記順が違う。実行順は以下1〜3の順だろう。2と3は逆転しても良いが、1は最初であるべきではないか？　4は最後であるべきだ。

1. `throw()`/`at()`/`judge()`のいずれかにより結果取得の制御方法を決定する
2. `pass()`により成功時の後続処理を書く
3. `fail()`により失敗時の後続処理を書く
4. `catch()`により`pass()`/`fail()`内で例外発生時の後続処理を書く

```
str.indexOf('abc', 'z')
.pass(index=>alert('成功!!: ', index))
.fail(res=>{
    document.body.appendChild(res.cause.constructor.name);
    document.body.appendChild(res.cause.message);
    throw new MyAppError(`APIエラーです。`, {cause:res.cause});
})
.catch(e=>{throw new MyAppError(`アプリケーションにとって予期せぬエラーです。`, {cause:e})});
.throw();
```

つまり上記は以下のような順序にすべきではないか？

```
str.indexOf('abc', 'z').throw()
.pass(index=>alert('成功!!: ', index)})
.fail(res=>{
    document.body.appendChild(res.cause.constructor.name);
    document.body.appendChild(res.cause.message);
    throw new MyAppError(`APIエラーです。`, {cause:res.cause});
})
.catch((e,isPass)=>{throw new MyAppError(`アプリケーションにとって予期せぬエラーです。`, {cause:e})});
```

たしかに実行順はその通りだし、読む時もそのほうが分かりやすい。でも実行するためには`throw()`/`at()`/`judge()`の動詞系メソッドは最後に書かねばならないはずでは？　少なくとも現行の実装ではそうなっている。`pass()`や`fail()`はあくまで実行完了後のコールバック関数を代入するだけのものであり、まだ結果取得方法を確定させてはいないのだから。

だとすると記述方法を変えたほうが良いか？　以下のように。

```
str.indexOf('abc', 'z').throw(res=>res
    .pass(index=>alert('成功!!: ', index)})
    .fail(res=>{
        document.body.appendChild(res.cause.constructor.name);
        document.body.appendChild(res.cause.message);
        throw new MyAppError(`APIエラーです。`, {cause:res.cause});
    })
    .catch((e,isPass)=>{throw new MyAppError(`アプリケーションにとって予期せぬエラーです。`, {cause:e})})
})
```

これにて順序は読みやすくなった。しかしこれは同時に、コールバック関数ネスト地獄の幕開けに思える。

この程度ならまだ許容範囲内だが、これが`addEventListener()`などのコールバック関数内に既に入ってるとか、しかもその中にある`try catch`, `if`, `for`などの中ともなれば、一体いくつネストすることになるやら。

だが、これなら例外処理まで含めて親側方向に`try catch`で囲まれた表記をすることなくDRYに書ける。そう考えると、どうせ親側`try catch`や`if`でネストするのだから、それを上記のように記述できたほうが読みやすいはずだ。

`throw()`の部分は`at()`や`judge()`に変えても動作すべきだ。裏を返せば、`judge()`/`throw()`/`at()`のどれであろうと、その結果は`pass()`/`fail()`で受け取ってルート分岐できるようにすべきだ。

```
str.indexOf('abc', 'z').at(res=>res
    .pass(index=>alert('成功!!: ', index)})
    .fail(res=>{
        document.body.appendChild(res.cause.constructor.name);
        document.body.appendChild(res.cause.message);
        throw new MyAppError(`APIエラーです。`, {cause:res.cause});
    })
    .catch((e,isPass)=>{throw new MyAppError(`アプリケーションにとって予期せぬエラーです。`, {cause:e})})
})
```
```
str.indexOf('abc', 'z').judge(res=>res
    .pass(index=>alert('成功!!: ', index)})
    .fail(res=>{
        document.body.appendChild(res.cause.constructor.name);
        document.body.appendChild(res.cause.message);
        throw new MyAppError(`APIエラーです。`, {cause:res.cause});
    })
    .catch((e,isPass)=>{throw new MyAppError(`アプリケーションにとって予期せぬエラーです。`, {cause:e})})
})
```

本来、`judge()`/`throw()`/`at()`には以下のような制御上の違いがある。

動詞|期待する挙動
----|------------
`judge()`|実装側に任せる（`throw()`/`at()`）
`throw()`|どんな失敗も`throw`する
`at()`|どんな失敗も`return`する

実際、それら動詞系メソッドの引数で渡したコールバック関数を実行し終わったら`throw()`では例外発生するし、`at()`では値返却するし、`judge()`では実装した通りに`throw()`なり`at()`なりをする。

そのことが動詞系メソッドの名前ですぐ分かるというのは重要だ。そしてやはりその動詞は、APIを実行したすぐ後に書くべきだろう。となると、結果取得後のルーティングは、その動詞メソッドの引数のコールバック関数で定義してもらう形が最善のように思える。

実行順に表記されていて分かりやすい。ネスト階層もそこまで多くない。結果取得後のコールバック関数設定は別に必須ではない。なので以下のようにも書ける。これらのことから、結果取得は以下のように外側で書くか、上記のように内側で書くか選べるようにするのが良いだろう。

```
// APIで問題あれば例外発生し中断する
const index = indexOf('abc', 'z').throw();
```
```
// 例外発生したら自前で制御したい
try {
    const index = str.indexOf('abc', 'z').throw();
} catch (e) {
    if (e instanceof JudgementFailure) {
        throw new MyAppError(`APIエラーです。`, {cause:e});
    } else {
        throw new MyAppError(`アプリケーションにとって予期せぬエラーです。`, {cause:e});
    }
}
```
```
// 成功／失敗でそれぞれ制御する。その後はjudge()/throw()/at()のどれを選ぶか次第で挙動が変わる
str.indexOf('abc', 'z').judge(res=>res
    .pass(index=>alert('成功!!: ', index)})
    .fail(res=>{
        document.body.appendChild(res.cause.constructor.name);
        document.body.appendChild(res.cause.message);
        throw new MyAppError(`APIエラーです。`, {cause:res.cause});
    })
    .catch((e,isPass)=>{throw new MyAppError(`アプリケーションにとって予期せぬエラーです。`, {cause:e})})
})
```
```
// Judgement利用APIをいくつも使うとネスト地獄になるため回避するなら以下のように書く
const res = str.indexOf('abc', 'z');
if (res.isFail) {return '失敗'}
const index = res.at();

const res1 = someJudgementApi1(index);
if (res1.isFail) {return '失敗'}
const ans1 = res1.at();

const res2 = someJudgementApi1(ans1);
if (res2.isFail) {return '失敗'}

```

## 価値・存在意義

judgement.jsの価値はあるか？　あるなら何か？

* 成否判定を真偽値で取得できる（`isPass`/`isFail`）
* 結果取得時の制御を選択できる（`at()`:`return`/`throw()`:`throw`/`judge()`:実装側次第(`return`/`throw`)）
* 結果取得後の制御記法を選択できる（二段構えによる同階層順次記述/コールバック関数によるネスト地獄）

上記によりエンドユーザは次のメリットを得られる。

* 結果の可読性向上（`-1`や`undefined`といったAPIごとに違うリテラル値に翻弄されずに済む）
* 制御権の獲得（APIの`return`や`throw`に決定権を奪われ態々`try catch`などを記述せずに済む）
* 保守性の堅持（可読性の高い統一されたAPIでDRYに書ける）
