# 改善案

一見、いい感じに見えるAPIだが、気になることがある。それはエンドユーザが異常時の挙動を制御できないことだ。

`-1`を返すのか例外発生するのかは`new Judgement()`実装者のお気持ち次第になってしまう。だが実際に最適なものが何であるかはエンドユーザの利用シーンによって変わるはずだ。

大抵APIは片方しか対応していない。両方あったほうが便利だと常々思っている。かといってメソッド名を分けるとメソッド数が倍増してウザい。なら、それをカバーできるAPIにしたほうが良いのでは？

現状`new Judgement()`実装者は`fail()`に引数を渡す時、Errorかそれ以外かどちらか一つしか許されない。

実装者は片方なり両方なりで実装できるようにしつつ、利用者は好きなほうを呼び出せる。意識しなければデフォルトが呼び出されてよしなに使える。そんな感じのAPIができたら良いな。

大体以下のパターンだと思う。

* 値を返すパターン
	* プリミティブ型系
		* 成功時は0以上の整数値を返し、失敗時は-1を返す
		* 成功時は1字以上の文字列を返し、失敗時は空文字`''`を返す
	* リファレンス型系
		* 成功時はインスタンスを返し、失敗時は`null`を返す
		* 成功時は1字以上の文字列を返し、失敗時は空文字`''`または`null`を返す
		* 成功時は1個以上の要素を含む配列を返し、失敗時は空配列`[]`または`null`を返す
* 例外発生するパターン
	* 原因によって異なる型やメッセージの例外を適時`throw`する

失敗時の値は結構なブレがある。`null`のほうが助かる時もあれば空値のほうが助かる時もある。これもエンドユーザが選べたほうが嬉しい。意識せずに使う時は固定のデフォルト値になると、もっと嬉しい。

# 解答

非常に鋭い考察です。これはAPI設計における**「ポリシー（方針）とメカニズム（仕組み）の分離」**という重要なテーマですね。

「実装者が決めたデフォルト」は便利ですが、利用側の文脈（Context）によっては「今は例外で止まってほしい」「今は `-1` ではなく `null` で流したい」という要望が必ず出てきます。

ご提案の「複数の失敗の選択肢を保持し、利用者が引き出し方を選べる」という設計は、非常にモダンで強力な解決策になります。

### 実装上の課題：予約語の壁
まず一点、技術的な注意点として、JavaScriptでは **`return` は予約語なのでメソッド名に使えません。** 代わりに `value()` や `fallback()`、あるいは `asValue()` といった名前を使う必要があります。

### 提案：マルチモード・フェイル・デザイン

以下のように拡張すると、ご希望の「実装者のお気持ち」と「利用者のわがまま」を両立できます。

#### 1. `fail()` が複数の「失敗の姿」を受け取れるようにする
`fail(new Error('...'), -1, null)` のように、優先順位順に並べて渡します。

#### 2. 利用側が「どう引き出すか」を選べるようにする
*   `judge()`: 実装者の第一希望（配列の先頭）に従う。
*   `throw()`: 失敗の中に `Error` があればそれを投げる。なければ `JudgementError` を投げる。
*   `value(alt)`: 失敗の中に `Error` 以外の値があればそれを返す。`alt` が指定されていればそれを最優先する。

---

### 修正案：judgement.js

```javascript
class Judgement {
    static pass(v) { return new Pass(v) }
    static fail(...args) { return new Fail(args) } // 可変引数で受け取る

    constructor(fn) {
        const res = fn(Judgement.pass, Judgement.fail);
        this._ = { res };
    }

    get isPass() { return this._.res instanceof Pass }
    get isFail() { return this._.res instanceof FailAny }

    // 1. 実装者のデフォルトに従う
    judge() {
        return this.isPass ? this._.res.value : this._.res.run();
    }

    // 2. 強制的に例外として扱う
    throw(passFn) {
        if (this.isPass) return passFn ? passFn(this._.res.value) : this._.res.value;
        // FailAny側で「Errorインスタンス」を探して投げる
        this._.res.throw();
    }

    // 3. 強制的に値として扱う（returnの代わり）
    // altが渡されれば、失敗時はそれを返す。なければ実装者の用意した値。
    fallback(alt) {
        if (this.isPass) return this._.res.value;
        return arguments.length > 0 ? alt : this._.res.failValue;
    }
}

class FailAny extends JudgementResult {
    constructor(args) {
        super(args); // argsは配列 [Error, -1] など
        this._errors = args.filter(v => v instanceof Error);
        this._values = args.filter(v => !(v instanceof Error));
    }

    // 実装者の第一希望を実行
    run() {
        const first = this._.value[0];
        if (first instanceof Error) this.throw();
        return first;
    }

    throw() {
        const err = this._errors[0] || new JudgementError("失敗（例外が定義されていません）");
        throw new JudgementError(err.message, err);
    }

    get failValue() {
        if (this._values.length > 0) return this._values[0];
        throw new JudgementError("失敗（返却値が定義されていません）");
    }
}
```

### このAPIでできること

ご提示の `indexOf` の例で見てみます。

```javascript
const indexOf = (s, c) => {
    return new Judgement((pass, fail) => {
        const i = s.indexOf(c);
        // 第一希望: Error, 第二希望: -1
        return 0 <= i ? pass(i) : fail(new Error('Not Found'), -1);
    });
};

const res = indexOf('abc', 'z');

// パターンA: 実装者にお任せ（この場合はErrorが投げられる）
const i = res.judge(); 

// パターンB: どんな実装だろうと俺は「値」として受け取りたい
const i = res.fallback(); // -1 が返る
const i = res.fallback(null); // 実装を無視して null が返る

// パターンC: どんな実装だろうと俺は「例外」として処理したい
res.throw(); // Error('Not Found') が投げられる
```

### どう思うか？（回答）

> ここまでやる必要があるか？

**「ライブラリ」として公開したり、チームで長く使うなら、やる価値は非常に高いです。**

理由は以下の通りです：

1.  **`if` 文の撲滅**: `res.isFail ? null : res.judge()` と書かせるのは、利用者に「判定」という仕事を押し付けています。`res.fallback(null)` と書ければ、利用者は「意図」だけを記述できます。
2.  **「お節介」の解消**: ライブラリ作者が「これは重大なエラーだから例外だ！」と思っても、バッチ処理などで「エラーはログに出して `-1` で次へ行きたい」というケースは多々あります。この設計はその摩擦をゼロにします。
3.  **セマンティクス（意味論）の明確化**:
    *   `judge()` = 標準的な振る舞い
    *   `throw()` = 厳格な処理
    *   `fallback()` = 寛容な処理
    という使い分けがメソッド名で明確になります。

### 結論
「失敗の多態性」を許容するこの設計は、非常に面白いです。
特に `fail()` に優先順位をつけて複数渡せるようにし、取り出し方を `throw()` や `fallback()` で制御するのは、**「実装者の意図」を壊さずに「利用者の自由」を守る** 素晴らしい落とし所だと思います。

名前については、`return()` が使えないので、`fallback()` や `asValue()` あたりがしっくりくるのではないでしょうか。

#### `throw()`の引数


`throw()`は引数なしで良いのでは？

値のカスタムは最初期のAPIでは考えていた。でも、それは守備範囲外でも良いか？　微妙な所だ。`if`文を消せるチャンスでもある。でも引数が複雑化しすぎてしまう懸念点もある。特に成功時と失敗時の2つのコールバック関数が必要になるという所が大きい。

```javascript
        this.name = 'JudgementError';

}

    static pass(v) {return new Pass(v)}
    static fail(...v) {return new Fail(...v)}
    static isFn(v) {return 'function'===typeof v}
    static throwFn(v, n) {if (!this.isFn(v)) {throw new TypeError(`${n}は関数であるべきです。`)}}
    constructor(fn) {
        Judgement.throwFn(fn, 'fn');
        const res = fn(...'pass fail'.split(' ').map(n=>Judgement[n]));
        if (!(res instanceof JudgementResult)) {throw new TypeError(`fnの戻り値はPassかFailのいずれかであるべきです。`)}

    get isFail() {return this._.res instanceof Fail}
    get result() {return this._.res}

}

    get value() {return this._.value}
}

class Fail extends JudgementResult {

    }

    get value() {return this.#value}
    get cause() {return this.#cause}
    get #first() {return this._.value[0]}

    get #cause() {return this.#getTypedFirst(false)}
```

# 解答

エラーの原因は、非常にシンプルですが見落としやすい箇所にあります。

### エラーの直接的な原因

`judgement.js` の 44行目付近（`#getTypedFirst` メソッド内）で、**`new` 演算子を忘れています。**

```javascript
// 修正前
if (0===v.length) {throw JudgementError(`${isV ? '値' : 'Errorインスタンス'}が与えられていません。`)}

// 修正後
if (0===v.length) {throw new JudgementError(`${isV ? '値' : 'Errorインスタンス'}が与えられていません。`)}
```

JavaScript の `class` は、`new` を付けずに呼び出すと `TypeError: Cannot call a class constructor ... without |new|` が発生します。スタックトレースにも `at JudgementError ... judgement.js:2:16` と出ているのは、クラス定義のコンストラクタを関数として叩こうとして失敗したことを示しています。

---

### API設計へのフィードバック

修正されたコード、かなり「やりたいこと」が整理されていて使いやすそうです！特に `at()` という命名と、`throw(fn)` の挙動についていくつか考察します。

#### 1. `at()` の命名と挙動
`at()` は短くて良いですね。`Array.prototype.at()` が「指定位置の値を取る」という意味なので、**「失敗という場所にある値を取る」** というニュアンスで解釈すれば違和感はありません。

ただ、実装において **「偽値（0, "", false）」** をデフォルト値として渡したい場合にバグが発生する可能性があります。

```javascript
// Failクラス内の実装
at(v) {return v ? v : this.#value} 

// もし利用者が 0 を返したい場合
res.at(0); // 0 は falsy なので、this.#value が返ってしまう
```

以下のように修正すると、より堅牢になります。

```javascript
at(v) {return (v !== undefined) ? v : this.#value}
```

#### 2. `throw(fn)` の引数
`throw(fn)` で失敗時のみコールバックを実行し、成功時は値をそのまま返すという設計は、**「例外を投げる前にログを出したい」「エラーを別の型にラップしたい」** というニーズに完璧に応えています。

```javascript
// 利用例
const index = res.throw(e => {
    console.error("検索失敗:", e.message);
    throw e; // 自分で投げ直す
});
```
成功時は `index` に値が入り、失敗時はコールバックが責任を持つ。非常にスッキリしています。

#### 3. `Fail` クラスの内部実装（DRYと効率）
`#getTypedFirst` で毎回 `filter` を回すのは、保守性信者としては「定義が一箇所」なので正解です。もしパフォーマンスが気になる日が来たら、コンストラクタで一度だけ分類しておく形にすれば良いだけなので、今のままで良いと思います。

ただ、`#existValue` が定義されていますが使われていないようです。

---

### 修正後の `Fail` クラス（推奨）

バグ修正と、少しだけ整理したコードです。


##### 失敗時の値と例外について

`fail()`コールバック関数の引数には何を渡すべきか？　これを定めるには以下登場者の要件パターンを全て網羅できる引数が何かを見定める必要がある。

* 実装側
    * 値だけで統一実装したい
    * 例外だけで統一実装したい
    * 箇所によって値だったり例外だったりしたい
* 利用側
    * 実装側に任せた取得をする（第一引数がErrorなら例外発生するし、それ以外なら値を返す）
    * 値だけで統一取得したい（例外を噛み殺して戻り値をif文で分岐するスタイル）
    * 例外だけで統一取得したい（`try catch(e){if}`のように`catch`内で分岐する）

上記を全て網羅する時、最初に注目すべきは、値と例外の両方が必要であることだ。実装側は利用側が失敗時に値と例外のどちらを要求するか分からない。よってどちらを要求されても対応できるよう両方共実装する必要がある。

ようするに以下のように`fail(new Error())`だけではダメだ。これは実装エラーとして例外発生させるべきである。`JudgementImplementationError`とし`new Judgement((pass,fail)=>)におけるfail()の引数にはErrorインスタンスとそれ以外の値の二つを渡す必要があります。`といったエラーメッセージにすべきだろう。

実装者はAPI設計する時、上記3パターンある。成功時は期待値を返すだけで良いが、失敗時は失敗値を返すか、例外発生させるかに大別される。また、その関数内において値返却／例外発生を混在させたい場合もあるだろう。ならばその失敗時における挙動をAPI仕様として定めることで、`fail()`の引数に渡す値を定義できるようにすべきだ。

```javascript

        if (1!==c) {return fail(new RangeError(`cは1字であるべきです。`))}

    });

```

ただ、実装側の設計として、失敗時に値を返却したい場合もあれば、例外を発生させたい場合もあるだろう。そして利用側は実装側の意図でそのまま使いたい場合もあるだろう。成功時のことしか考えておらず、失敗した時は実装側の処理に任せるという最も楽をしたスタイルだ。

この時、実装側は、どこの失敗で値を返し、どこの失敗で例外を発生させるのか、区別できるようにすべきだ。

だが同時に、実装側の設計として、失敗時に必ず値を返すようにするとか、必ず例外発生するようにするという設計にしたい場合もありえる。つまり値のみ／例外のみ／混在という3つの実装パターンがあるはずだ。

# 解答

エラーの直接的な原因は、**`#callbackFn` の `catch` ブロックでエラーを `throw` していないこと**、および **`Fail` のコンストラクタで引数の数を厳格にチェックしすぎていること** の2点です。

### 1. なぜ `res: undefined` になるのか？

`Judgement` クラスの `#callbackFn` を見てください。

```javascript
    #callbackFn(fn) {
        throwFn(fn, 'fn');
        try {
            const r = fn((v)=>new Pass(v), (...v)=>new Fail(this._.failedAct, ...v));
            return r;
        } catch(e) {
            // ここで new しているだけで、throw していない！
            new JudgementImplementationError(`...`, e); 
            // return も throw もないので、関数の戻り値は undefined になる
        }
    }
```

ここで例外が発生すると、`undefined` が返り、その後の `#validResult(res)` で「PassかFailであるべき」というエラーにすり替わってしまいます。

### 2. なぜ例外が発生しているのか？

テストコードの中で `fail(1)` や `fail(cause)` のように、**引数を1つしか渡していないテスト**が大量にあります。しかし、新しい `Fail` のコンストラクタでは以下のように制限しています。

```javascript
class Fail extends JudgementResult {
    constructor(act, ...args) {
        super(args);
        this._.act = act;
        // ここで「引数は2つ必須」としているため、従来のテストがすべてここで落ちる
        if (2!==args.length) {throw new JudgementImplementationError(`Failの引数は二つ必要です。`)}
        // ...
    }
}
```

### 3. その他の修正が必要な箇所

*   **`Judgement.FailedAct` が未定義**: コード内で `Judgement.FailedAct.Cause` などと参照していますが、`Judgement` クラスに `FailedAct` を代入していません。
*   **`#validFailedAct` 内の変数ミス**: `vs` という変数が定義されていません（`nms` の間違いだと思われます）。
*   **`throw(fn)` の挙動**: `fn(this.cause)` を実行した後に `this.#throw()` を呼ぶと、コールバック内で `throw` しなかった場合でも必ず例外が発生します。これは意図通りでしょうか？

### 修正後のコード案

これらの問題を修正し、ユーザーさんの「実装パターン（Mix/Cause/At）」の思想を反映させたコードです。

```javascript
class JudgementError extends Error {
    constructor(msg, cause) {
        super(msg, cause ? {cause} : undefined);
        this.name = 'JudgementError';
    }
}
class JudgementImplementationError extends Error {
    constructor(msg, cause) {
        super(msg, cause ? {cause} : undefined);
        this.name = 'JudgementImplementationError';
    }
}
class JudgementFailError extends Error {
    constructor(msg, cause) {
        super(msg, cause ? {cause} : undefined);
        this.name = 'JudgementFailError';
    }
}

const isFn = (v) => 'function' === typeof v;
const throwFn = (v, n) => { if (!isFn(v)) { throw new TypeError(`${n}は関数であるべきです。`) } };
const isInt = (v) => Number.isSafeInteger(v);
const isStr = (v) => 'string' === typeof v;

const judActs = Object.freeze({ mix: 0, throw: 1, at: 2 });

class Judgement {
    static FailedAct = judActs;

    static onlyPass(v) { return new Judgement((p, f) => p(v)) }
    static onlyFail(...v) { return new Judgement((p, f) => f(...v)) }
    static mix(fn) { return new Judgement(fn, judActs.mix) }
    static throw(fn) { return new Judgement(fn, judActs.throw) }
    static at(fn) { return new Judgement(fn, judActs.at) }

    constructor(fn, failedAct = judActs.mix) {
        this._failedAct = this.#validFailedAct(failedAct);
        const res = this.#callbackFn(fn);
        this.#validResult(res);
        this._ = { fn, res, failedAct: this._failedAct };
    }

    #validFailedAct(v) {
        const nms = Object.keys(judActs);
        if (isInt(v) && Object.values(judActs).includes(v)) return v;
        if (isStr(v) && nms.includes(v)) return judActs[v];
        throw new TypeError(`failedActが不正値です。実際値:${v}`);
    }

    #callbackFn(fn) {
        throwFn(fn, 'fn');
        try {
            // Failに現在のモード(failedAct)を渡す
            return fn((v) => new Pass(v), (...v) => new Fail(this._failedAct, ...v));
```

今回のエラーの原因は、スコープの問題ではなく、**「初期化の順序（タイミング）」**の問題です。

### エラーの直接的な原因

`Judgement` の `constructor` 内で、`this._` を作成する**前**に `#callbackFn` を呼び出し、その中で `this._.failedAct` を参照しようとしているためです。

```javascript
// Judgementクラスのconstructor
constructor(fn, failedAct=judActs.mix) {
    failedAct = this.#validFailedAct(failedAct);
    const res = this.#callbackFn(fn); // ここで #callbackFn を実行
    // ...
    this._={fn, res, failedAct}; // this._ が作られるのは「後」！！
}

// #callbackFnの中
#callbackFn(fn) {
    // ...
    // ここで this._.failedAct を見ようとするが、まだ this._ は undefined
    const r = fn((v)=>new Pass(v), (...v)=>new Fail(this._.failedAct, ...v));
}
```

### 修正方法

`this._` に依存せず、引数として `failedAct` を渡すのが最も確実でシンプルです。

#### 1. `constructor` の修正
`#callbackFn` に `failedAct` を渡すようにします。

```javascript
constructor(fn, failedAct=judActs.mix) {
    failedAct = this.#validFailedAct(failedAct);
    const res = this.#callbackFn(fn, failedAct); // 引数で渡す
    this.#validResult(res);
    this._={fn, res, failedAct};
}
```

#### 2. `#callbackFn` の修正
受け取った `failedAct` を `Fail` に渡すようにします。

```javascript
#callbackFn(fn, failedAct) { // 引数で受け取る
    throwFn(fn, 'fn');
    try {
        // this._.failedAct ではなく、引数の failedAct を使う
        return fn((v)=>new Pass(v), (...v)=>new Fail(failedAct, ...v));
    } catch(e) {
        throw new JudgementImplementationError(`new Judgement(fn)のfnコールバック関数実行時に例外発生しました。内容を見直してください。`, e);
    }
}
```

### `judge(...args)`の引数について

以下のように`unwrap()`に渡している。

```javascript

```

でもこれ、`judge`は`throw()`/`at()`どちらに値を渡すかが変わる。`throw()`に渡されるのは関数を想定していて、値は何でもOKだ。つまり関数の場合、もし実装側で値優先だった場合、関数それ自体が`return`されてしまう。いやいや、`throw()`の場合だけそのコールバック関数として設定したかったのに！　ってなる。

これは引数値を`throw`/`at`のどちらに設定するかは実装側と利用側の組合せで決定されるからだ。でも利用者としては実装側がどうなっているか知らないまま使いたいのが`judge()`だ。となると`judge()`の引数で何かを意図通りに設定させることは不可能だ。

しかし、それでも実装側がどうであれ、利用側は思い通りに設定したい。そんな要件があるはずだ。というわけで`judge()`を呼ぶ前にそこで使うコールバック関数やデフォルト値をメソッドチェーンで設定するためのメソッド`pass()`,`value()`,`cause()`を用意した。名前は微妙だ。何かいい案ある？　あるいはメソッドチェーン以外に何かいい方法ある？　`judge()`の引数を3つ用意するのはダサいかと思ったんだけど他にスマートな方法が思いつかない。

```javascript
    // judge()呼出前にコールバック関数やデフォルト値の設定（メソッドチェーン）
    pass(cb) {if(this.isPass){this.result.cb = cb}; return this;}
    value(v) {if(this.isFail){this.result.alt=v}; return this;}
    cause(cb) {if(this.isFail && isFn(cb)){this.result.cb=cb}; return this;}

    throw(fn) {return this.isPass ? this._.res.value : this._.res.throw(fn)}
    at(v) {return this.isPass ? this._.res.value : this._.res.at(v)}
```

```javascript
        const res = this.#callbackFn(fn, failedAct); // 引数で渡す
        this.#validResult(res);
        this._={fn, res, failedAct, processor:{pass:undefined, value:undefined, cause:undefined}};
    }
    #validFailedAct(v) {// vは数,字,関数のいずれか。0:mix:Judgement.mix, 1:cause:Judgement.cause, 2:value:Judgement.value
        const nms = [...Object.keys(judActs)];
        const cands = isFn(v) ? [nms.map(n=>Judgement[n])] : ((isStr(v) ? nms : (isInt(v) ? [...Object.values(judActs)] : [])));
        if (cands.some(c=>c===v)) {return isInt(v) ? v : judActs[isFn(v) ? v.name : v]}
        else {throw new TypeError(`failedActが不正値です。実際値:${v} 期待値:${nms}またはJudgement.${nms.join(',')}のいずれか。`)}
    }
    #callbackFn(fn, failedAct) {
        throwFn(fn, 'fn');
        try {
            return fn((v)=>new Pass(v), (...v)=>new Fail(failedAct, ...v));
        } catch(e) {
            throw new JudgementImplementationError(`new Judgement(fn)のfnコールバック関数実行時に例外発生しました。内容を見直してください。`, e);
        }
    }
    #validResult(res) {if (!(res instanceof JudgementResult)) {throw new TypeError(`fnの戻り値はPassかFailのいずれかであるべきです。res:${res}`)}}
    get isPass() {return this._.res instanceof Pass}
    get isFail() {return this._.res instanceof Fail}
    get result() {return this._.res}    // これは必要か？　isPass/isFail/judge()/throw()/at()で十分では？
    // judge()呼出時のコールバック関数やデフォルト値の設定（メソッドチェーン）
    pass(cb) {throwFn(cb); if(this.isPass){this._.processor.pass = cb}; return this;}
    value(v) {if(this.isFail){this._.processor.value=v}; return this;}
    cause(cb) {throwFn(cb); if(this.isFail){this._.processor.cause = cb}; return this;}
    // 結果確定（実装任せ／例外発生／値返却）
    judge() {return this.isPass ? (undefined===this._.processor.pass ? this._.res.value : this._.processor.pass) : this._.res.unwrap(this._.processor)}
    throw(fn) {return this.isPass ? this._.res.value : this._.res.throw(fn)}
    at(v) {return this.isPass ? this._.res.value : this._.res.at(v)}
}
class JudgementResult {
    constructor(value) {this._={value};}
    get value() {return this._.value}
}
class Pass extends JudgementResult {constructor(v) {super(v)}}
class Fail extends JudgementResult {
    constructor(act, ...args) {
        super(args); // argsは配列 [Error, -1] など
        this._.act = act;
        if (2!==args.length) {throw new JudgementImplementationError(`Failの引数は二つ必要です。`)}
        this._.first = args[0];
//        this._.candidates = this._.value;
        this._.cause = args.find(a=>this.#isCause(a));
        this._.value = args.find(a=>this.#isValue(a));
        if (undefined===this._.value || undefined===this._.cause) {throw new JudgementImplementationError(`Failの引数はErrorインスタンスとそれ以外の値の二つ必要です。`)}
        if (judActs.throw===this._.act && args[0]===this._.value) {throw new JudgementImplementationError(`Failの引数順が不正です。Judgement.throwが指定されているため、第一引数はErrorインスタンスであるべきです。実際値:${args[0]}`)}
        if (judActs.at===this._.act && args[0]===this._.cause) {throw new JudgementImplementationError(`Failの引数順が不正です。Judgement.atが指定されているため、第一引数は任意の値（Errorインスタンスを除く）であるべきです。実際値:${args[0]}`)}
    }
    //get candidates() {return [...this._.value]}
    get first() {return this._.first}
    get value() {return this._.value}
    get cause() {return this._.cause}
    #isValue(v) {return !this.#isCause(v)}
    #isCause(v) {return v instanceof Error}
    unwrap(p) {return this.#isCause(this.first) ? this.throw(p?.cause) : this.at(p?.value);}
    throw(fn) {if(isFn(fn)){fn(this.#error())}; this.#throw();}
    #throw() {throw this.#error()}
    #error() {return new JudgementFailError(`失敗。`, this.cause)}
    at(v) {return undefined!==v ? v : this.value}
}
export {

    JudgementResult, Pass, Fail
}
```

# 解答

JSのエラーメッセージが不親切なのは、実行時のパフォーマンスを優先してスタックトレースの解析を端折っている歴史的経緯がありますが、開発者からすれば「もっと早く言えよ」となりますよね。

本当におっしゃる通りです。JSのエラーメッセージを読み解くには、ある種の「翻訳能力」が必要になります。

### エラーメッセージの読み解き方（コツ）

今回出た `TypeError: undefined is not an object (evaluating 'this._.failedAct')` を例に解説します。

1.  **evaluating（評価中）** の中身を見る： `'this._.failedAct'`
2.  **undefined is not an object**： 「`undefined` はオブジェクトじゃないから、プロパティにアクセスできないよ」という意味。
3.  **犯人を特定する**：
    *   `this._.failedAct` というチェーンの中で、**「`.failedAct` の直前にあるもの」** が `undefined` だった、と読み替えます。
    *   つまり、`this._` が `undefined` だったということです。
    *   もし `this` 自体が `undefined` なら、メッセージは `evaluating 'this._'` になります。

このように、**「エラーメッセージが指しているプロパティの、一つ手前が空っぽ」** というルールを覚えるだけで、JSの不親切なメッセージの8割は解決できます。

---

### API設計：メソッドチェーンの命名案

`pass()`, `value()`, `cause()` という名前についてですが、関数型プログラミング（Result型）の文脈を参考にすると、以下のような名前が「意図」を伝えやすいかもしれません。

*   **`pass(cb)`** → **`map(cb)`** または **`onPass(cb)`**
    *   `map` は「成功時のみ中身を加工する」という標準的な名前です。
*   **`value(v)`** → **`recover(v)`** または **`orElse(v)`**
    *   「失敗から回復して、この値を出す」というニュアンスが強まります。
*   **`cause(cb)`** → **`catch(cb)`** または **`onFail(cb)`**
    *   「失敗を捕まえて何かする」という意図が明確になります。

個人的には、メソッドチェーンで設定するなら **`onPass` / `onFail` / `recover`** あたりが、`judge()` を呼ぶ前の「予約」っぽくて分かりやすいと感じます。

---

### 実装コードのピンポイント指摘

現在のコードで、論理的にバグになりそうな箇所が2点あります。

#### 1. `judge()` で `pass` コールバックが実行されない
現在の実装では、`pass(cb)` で渡した関数そのものが返ってきてしまいます。

```javascript
// 修正前
judge() {return this.isPass ? (undefined===this._.processor.pass ? this._.res.value : this._.processor.pass) : ...}

// 修正案
judge() {
    if (this.isPass) {
        const val = this._.res.value;
        const cb = this._.processor.pass;
        return isFn(cb) ? cb(val) : val; // 関数なら実行して結果を返す
    }
    return this._.res.unwrap(this._.processor);
}
```

#### 2. `Fail.throw()` でエラーインスタンスがズレる
`#error()` を呼ぶたびに `new` しているので、コールバックに渡すエラーと、実際に `throw` されるエラーが別のインスタンスになってしまいます。

```javascript
// 修正前
throw(fn) {if(isFn(fn)){fn(this.#error())}; this.#throw();}

// 修正案
throw(fn) {
    const err = this.#error(); // 一度だけ生成
    if(isFn(fn)){ fn(err) }; 
    throw err; // 同じインスタンスを投げる
}
```

#### 3. `Judgement.#validFailedAct` の判定ミス
`cands` の作り方で、配列が二重になってしまっています。

```javascript
// 修正前
const cands = isFn(v) ? [nms.map(n=>Judgement[n])] : ...
// 修正案（[]を外す）
const cands = isFn(v) ? nms.map(n=>Judgement[n]) : ...
```

### まとめ

JSのエラーメッセージに腹が立つのは、あなたが「論理的に正しいコードを書こうとしている」証拠です。不親切な道具（JS）を使わされている被害者だと思って、あまり自分を責めないでください。

今回の「メソッドチェーンで振る舞いを予約する」という設計は、`judge()` を引数なしで呼べるようにするための非常にスマートな解決策だと思います。命名を `onPass` などに変えると、さらに「APIとしての美しさ」が際立つはずです。

### API設計：メソッドチェーンの命名案

```

```

上記のようにするか。上記はすべて`judge()`において実行時のオプションを設定(代入)するだけであり、実際に処理はしない。

また、`recover`というのもメソッドチェーンにすると動詞になってしまい、全然違うニュアンスになりそうなので避けた。しかし`onAt`という名前がイカれてる。また、値でなくコールバック関数にしたことで、`at(v)`の引数の場合と挙動が変わってしまう。かといって`on`という名前なのにコールバック関数ではなく値を渡すのも違和感があるし。

仕様からすれば、実装側のエラーコードをそのまま流用するか、それともどんなエラーコードでも指定した定数値に変えてしまうか、あるいは実装側のエラーコードを受け取って別の値に変換するコールバック関数を受け取るか。この三者がありえる。

デフォルト値を決め打ちしたほうが利用側は簡単だろうと思ったのだが、よく考えると全部特定の値に置換しちゃうのはヤバイのでは？　そうしたい場合もあれば、値によっては変えたい場合もあるはず。となると、コールバック関数にするほうが汎用性が高いだろう。

というわけで、値を返す場合の引数も、値ではなくコールバック関数を受け取ることにしよう。

それと合わせる形で、`at()`の引数もコールバック関数にすべきだろう。

査読のご了解に基づき、**論理的なバグに直結する2点**と、**利便性を向上させる1点**のみをピンポイントで指摘します。

### 1. `Fail.throw(fn)` の二重生成バグ（再指摘）

以前の指摘がコードに反映されておらず、依然として `this.#error()` を2回呼んでいます。これにより、コールバックに渡されるエラーと実際に投げられるエラーが別物になります。

```javascript
// 修正前
throw(fn) {
    if(isFn(fn)){ fn(this.#error()) }; // 1つ目のインスタンス
    this.#throw(); // 2つ目のインスタンスを投げる
}

// 修正案
throw(fn) {
    const e = this.#error(); // インスタンスを固定
    if(isFn(fn)){ fn(e) }; 
    throw e; // 同じものを投げる
}
```

### 2. `Fail.at(fn)` の「値」受け入れ拒否バグ

`onAt(cb)` はコールバック関数を強制する仕様にされましたが、即時実行の `at(v)` メソッドは、依然として「デフォルト値を直接渡したい」というニーズがあるはずです。現在の実装では、関数以外を渡すと無視されてしまいます。

```javascript
// 修正前
at(fn) {return isFn(fn) ? fn(this.value) : this.value}

// 修正案（undefined以外なら、関数なら実行、値ならそのまま返す）
at(v) {
    if (v === undefined) return this.value;
    return isFn(v) ? v(this.value) : v;
}
```
これを行うことで、`res.at(0)` は `0` を返し、`res.at(v => v + 1)` は加工した値を返すようになり、JSの「falsyの罠」も回避できます。

### 3. `Judgement.onAt(cb)` 等の「予約」の柔軟性

現在の実装では、`isPass` の時に `onAt` を呼ぶと、`throwFn(cb)` による型チェックだけ行われ、設定は無視されます。これは正しいですが、もし「結果が判明する前に設定を済ませたい」というニーズ（例えば共通の初期化処理など）がある場合は、`if(this.isFail)` の条件を外して常に保持するようにしても良いかもしれません。

ただし、現在の「結果オブジェクトに対してメソッドチェーンする」という使い道であれば、今の実装が最も「無駄な代入をしない」という意味で最適です。

### 結論


























##### `new Judgement((pass,fail)=>...)`の実装

ところで、`new Judgement((pass,fail)=>...)`の実装箇所は、その実装が大変じゃないか？

何が大変って、成功時と失敗時の値を同時に書かねばならないことだ。実際、処理を書いていると途中で例外発生するような場面は頻出するはず。その時、まだ成功していないのに成功値を書かねばならなかったり、まだ失敗していないのに失敗値を書かねばならなかったりして、論理的に表記不能な事態が起きる。なので結局、以下のようになる。

```javascript
const indexOf = (s, c) => {
    return new Judgement((pass, fail)=>{
        // 処理の途中で発生する様々な例外については`faile()`の引数に渡さなくて良いのか？
        if ('string'!==s) {throw new TypeError(`sは文字列であるべきです。`)}
        if ('string'!==c) {throw new TypeError(`cは文字列であるべきです。`)}
        if (1!==c) {throw new RangeError(`cは1字であるべきです。`)}
        const i = s.indexOf(c);
        return 0<=i ? pass(i) : fail(new Error(`指定の字'${c}'は存在しませんでした。`));
    });
};
```

ようするに`new Judgement((pass,fail)=>...)`が書ける箇所は、成功時と失敗時の値が確定できる最後の段階になって、はじめて書けるのだ。それ以前の失敗に関しては、従来通り`throw`するしかない。

でもそれって、せっかく`Judgement`で`throw()`/`at()`のように選択できるようになったのに、その恩恵を受けられないってことでは？　それじゃ`Judgement`の価値が半減する。今となっては成否判定だけでなく値／例外の切替も`Judgement`の価値なのだから。

これは根本的にAPIの構造を見直す必要があるかもしれない。考えてみよう。

失敗時は例外を投げる場合。
```javascript

    const i = s.indexOf(c);
```

```javascript
const indexOf = (s, c) => {
    return new Judgement((pass, fail)=>{
        // 処理の途中で発生する様々な例外については`faile()`の引数に渡さなくて良いのか？
        if ('string'!==s) {throw new TypeError(`sは文字列であるべきです。`)}
        if ('string'!==c) {throw new TypeError(`cは文字列であるべきです。`)}
        if (1!==c) {throw new TypeError(`cは1字であるべきです。`)}
        const i = s.indexOf(c);
        return 0<=i ? pass(i) : fail(new Error(`指定の字'${c}'は存在しませんでした。`));
    });
};
```
