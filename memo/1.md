# 改善案

一見、いい感じに見えるAPIだが、気になることがある。それはエンドユーザが異常時の挙動を制御できないことだ。

`-1`を返すのか例外発生するのかは`new Judgement()`実装者のお気持ち次第になってしまう。だが実際に最適なものが何であるかはエンドユーザの利用シーンによって変わるはずだ。

大抵APIは片方しか対応していない。両方あったほうが便利だと常々思っている。かといってメソッド名を分けるとメソッド数が倍増してウザい。なら、それをカバーできるAPIにしたほうが良いのでは？

現状`new Judgement()`実装者は`fail()`に引数を渡す時、Errorかそれ以外かどちらか一つしか許されない。

実装者は片方なり両方なりで実装できるようにしつつ、利用者は好きなほうを呼び出せる。意識しなければデフォルトが呼び出されてよしなに使える。そんな感じのAPIができたら良いな。

大体以下のパターンだと思う。

* 値を返すパターン
	* プリミティブ型系
		* 成功時は0以上の整数値を返し、失敗時は-1を返す
		* 成功時は1字以上の文字列を返し、失敗時は空文字`''`を返す
	* リファレンス型系
		* 成功時はインスタンスを返し、失敗時は`null`を返す
		* 成功時は1字以上の文字列を返し、失敗時は空文字`''`または`null`を返す
		* 成功時は1個以上の要素を含む配列を返し、失敗時は空配列`[]`または`null`を返す
* 例外発生するパターン
	* 原因によって異なる型やメッセージの例外を適時`throw`する

失敗時の値は結構なブレがある。`null`のほうが助かる時もあれば空値のほうが助かる時もある。これもエンドユーザが選べたほうが嬉しい。意識せずに使う時は固定のデフォルト値になると、もっと嬉しい。

# 解答

非常に鋭い考察です。これはAPI設計における**「ポリシー（方針）とメカニズム（仕組み）の分離」**という重要なテーマですね。

「実装者が決めたデフォルト」は便利ですが、利用側の文脈（Context）によっては「今は例外で止まってほしい」「今は `-1` ではなく `null` で流したい」という要望が必ず出てきます。

ご提案の「複数の失敗の選択肢を保持し、利用者が引き出し方を選べる」という設計は、非常にモダンで強力な解決策になります。

### 実装上の課題：予約語の壁
まず一点、技術的な注意点として、JavaScriptでは **`return` は予約語なのでメソッド名に使えません。** 代わりに `value()` や `fallback()`、あるいは `asValue()` といった名前を使う必要があります。

### 提案：マルチモード・フェイル・デザイン

以下のように拡張すると、ご希望の「実装者のお気持ち」と「利用者のわがまま」を両立できます。

#### 1. `fail()` が複数の「失敗の姿」を受け取れるようにする
`fail(new Error('...'), -1, null)` のように、優先順位順に並べて渡します。

#### 2. 利用側が「どう引き出すか」を選べるようにする
*   `judge()`: 実装者の第一希望（配列の先頭）に従う。
*   `throw()`: 失敗の中に `Error` があればそれを投げる。なければ `JudgementError` を投げる。
*   `value(alt)`: 失敗の中に `Error` 以外の値があればそれを返す。`alt` が指定されていればそれを最優先する。

---

### 修正案：judgement.js

```javascript
class Judgement {
    static pass(v) { return new Pass(v) }
    static fail(...args) { return new Fail(args) } // 可変引数で受け取る

    constructor(fn) {
        const res = fn(Judgement.pass, Judgement.fail);
        this._ = { res };
    }

    get isPass() { return this._.res instanceof Pass }
    get isFail() { return this._.res instanceof FailAny }

    // 1. 実装者のデフォルトに従う
    judge() {
        return this.isPass ? this._.res.value : this._.res.run();
    }

    // 2. 強制的に例外として扱う
    throw(passFn) {
        if (this.isPass) return passFn ? passFn(this._.res.value) : this._.res.value;
        // FailAny側で「Errorインスタンス」を探して投げる
        this._.res.throw();
    }

    // 3. 強制的に値として扱う（returnの代わり）
    // altが渡されれば、失敗時はそれを返す。なければ実装者の用意した値。
    fallback(alt) {
        if (this.isPass) return this._.res.value;
        return arguments.length > 0 ? alt : this._.res.failValue;
    }
}

class FailAny extends JudgementResult {
    constructor(args) {
        super(args); // argsは配列 [Error, -1] など
        this._errors = args.filter(v => v instanceof Error);
        this._values = args.filter(v => !(v instanceof Error));
    }

    // 実装者の第一希望を実行
    run() {
        const first = this._.value[0];
        if (first instanceof Error) this.throw();
        return first;
    }

    throw() {
        const err = this._errors[0] || new JudgementError("失敗（例外が定義されていません）");
        throw new JudgementError(err.message, err);
    }

    get failValue() {
        if (this._values.length > 0) return this._values[0];
        throw new JudgementError("失敗（返却値が定義されていません）");
    }
}
```

### このAPIでできること

ご提示の `indexOf` の例で見てみます。

```javascript
const indexOf = (s, c) => {
    return new Judgement((pass, fail) => {
        const i = s.indexOf(c);
        // 第一希望: Error, 第二希望: -1
        return 0 <= i ? pass(i) : fail(new Error('Not Found'), -1);
    });
};

const res = indexOf('abc', 'z');

// パターンA: 実装者にお任せ（この場合はErrorが投げられる）
const i = res.judge(); 

// パターンB: どんな実装だろうと俺は「値」として受け取りたい
const i = res.fallback(); // -1 が返る
const i = res.fallback(null); // 実装を無視して null が返る

// パターンC: どんな実装だろうと俺は「例外」として処理したい
res.throw(); // Error('Not Found') が投げられる
```

### どう思うか？（回答）

> ここまでやる必要があるか？

**「ライブラリ」として公開したり、チームで長く使うなら、やる価値は非常に高いです。**

理由は以下の通りです：

1.  **`if` 文の撲滅**: `res.isFail ? null : res.judge()` と書かせるのは、利用者に「判定」という仕事を押し付けています。`res.fallback(null)` と書ければ、利用者は「意図」だけを記述できます。
2.  **「お節介」の解消**: ライブラリ作者が「これは重大なエラーだから例外だ！」と思っても、バッチ処理などで「エラーはログに出して `-1` で次へ行きたい」というケースは多々あります。この設計はその摩擦をゼロにします。
3.  **セマンティクス（意味論）の明確化**:
    *   `judge()` = 標準的な振る舞い
    *   `throw()` = 厳格な処理
    *   `fallback()` = 寛容な処理
    という使い分けがメソッド名で明確になります。

### 結論
「失敗の多態性」を許容するこの設計は、非常に面白いです。
特に `fail()` に優先順位をつけて複数渡せるようにし、取り出し方を `throw()` や `fallback()` で制御するのは、**「実装者の意図」を壊さずに「利用者の自由」を守る** 素晴らしい落とし所だと思います。

名前については、`return()` が使えないので、`fallback()` や `asValue()` あたりがしっくりくるのではないでしょうか。

#### `throw()`の引数


上記は`throw()`だけが成功時の値をカスタムできるようになっていることに違和感がある。他の`judge()`や`return()`では成功時の値をカスタムするコールバック関数はないのに。また、失敗時の値をカスタムするコールバック関数はなくても良いのか？　そういう点でも違和感がすごい。

`throw()`は引数なしで良いのでは？

値のカスタムは最初期のAPIでは考えていた。でも、それは守備範囲外でも良いか？　微妙な所だ。`if`文を消せるチャンスでもある。でも引数が複雑化しすぎてしまう懸念点もある。特に成功時と失敗時の2つのコールバック関数が必要になるという所が大きい。

```javascript

    constructor(msg, cause) {
        super(msg, cause ? {cause} : undefined);
        this.name = 'JudgementError';
    }
}

    static pass(v) {return new Pass(v)}
    static fail(...v) {return new Fail(...v)}
    static isFn(v) {return 'function'===typeof v}
    static throwFn(v, n) {if (!this.isFn(v)) {throw new TypeError(`${n}は関数であるべきです。`)}}
    constructor(fn) {
        Judgement.throwFn(fn, 'fn');
        const res = fn(...'pass fail'.split(' ').map(n=>Judgement[n]));
        if (!(res instanceof JudgementResult)) {throw new TypeError(`fnの戻り値はPassかFailのいずれかであるべきです。`)}
        this._={fn:fn, res:res};
    }
    get isPass() {return this._.res instanceof Pass}
    get isFail() {return this._.res instanceof Fail}
    get result() {return this._.res}
    judge() {return this.isPass ? this._.res.value : this._.res.unwrap()}
    throw(fn) {return this.isPass ? this._.res.value : (Judgement.isFn(fn) ? fn(this._.res.cause) : this._.res.throw())}
    at(v) {return this.isPass ? this._.res.value : (undefined===v ? this._.res.at() : v)}
}

    constructor(value) {this._={value}}
    get value() {return this._.value}
}
class Pass extends JudgementResult {constructor(v) {super(v)}}
class Fail extends JudgementResult {
    constructor(...args) {
        super(args); // argsは配列 [Error, -1] など
        if (0===args.length) {throw new TypeError(`Failの引数は一つ以上必要です。`)}
    }
    get candidates() {return [...this._.value]}
    get value() {return this.#value}
    get cause() {return this.#cause}
    get #first() {return this._.value[0]}
    get #value() {return this.#getTypedFirst(true)}
    get #cause() {return this.#getTypedFirst(false)}

    get #existValue() {return this._.value.filter(v=>this.#isValue(v))}
    unwrap() {return this.#isCause(this.#first) ? this.throw() : this.at()}
```

# 解答

エラーの原因は、非常にシンプルですが見落としやすい箇所にあります。

### エラーの直接的な原因

`judgement.js` の 44行目付近（`#getTypedFirst` メソッド内）で、**`new` 演算子を忘れています。**

```javascript
// 修正前
if (0===v.length) {throw JudgementError(`${isV ? '値' : 'Errorインスタンス'}が与えられていません。`)}

// 修正後
if (0===v.length) {throw new JudgementError(`${isV ? '値' : 'Errorインスタンス'}が与えられていません。`)}
```

JavaScript の `class` は、`new` を付けずに呼び出すと `TypeError: Cannot call a class constructor ... without |new|` が発生します。スタックトレースにも `at JudgementError ... judgement.js:2:16` と出ているのは、クラス定義のコンストラクタを関数として叩こうとして失敗したことを示しています。

---

### API設計へのフィードバック

修正されたコード、かなり「やりたいこと」が整理されていて使いやすそうです！特に `at()` という命名と、`throw(fn)` の挙動についていくつか考察します。

#### 1. `at()` の命名と挙動
`at()` は短くて良いですね。`Array.prototype.at()` が「指定位置の値を取る」という意味なので、**「失敗という場所にある値を取る」** というニュアンスで解釈すれば違和感はありません。

ただ、実装において **「偽値（0, "", false）」** をデフォルト値として渡したい場合にバグが発生する可能性があります。

```javascript
// Failクラス内の実装
at(v) {return v ? v : this.#value} 

// もし利用者が 0 を返したい場合
res.at(0); // 0 は falsy なので、this.#value が返ってしまう
```

以下のように修正すると、より堅牢になります。

```javascript
at(v) {return (v !== undefined) ? v : this.#value}
```

#### 2. `throw(fn)` の引数
`throw(fn)` で失敗時のみコールバックを実行し、成功時は値をそのまま返すという設計は、**「例外を投げる前にログを出したい」「エラーを別の型にラップしたい」** というニーズに完璧に応えています。

```javascript
// 利用例
const index = res.throw(e => {
    console.error("検索失敗:", e.message);
    throw e; // 自分で投げ直す
});
```
成功時は `index` に値が入り、失敗時はコールバックが責任を持つ。非常にスッキリしています。

#### 3. `Fail` クラスの内部実装（DRYと効率）
`#getTypedFirst` で毎回 `filter` を回すのは、保守性信者としては「定義が一箇所」なので正解です。もしパフォーマンスが気になる日が来たら、コンストラクタで一度だけ分類しておく形にすれば良いだけなので、今のままで良いと思います。

ただ、`#existValue` が定義されていますが使われていないようです。

---

### 修正後の `Fail` クラス（推奨）

バグ修正と、少しだけ整理したコードです。

```javascript
class Fail extends JudgementResult {
    constructor(...args) {
        super(args);
        if (0 === args.length) { throw new TypeError(`Failの引数は一つ以上必要です。`) }
    }
    get candidates() { return [...this._.value] }
    get value() { return this.#getTypedFirst(true) }
    get cause() { return this.#getTypedFirst(false) }
    
    get #first() { return this._.value[0] }

    #getTypedFirst(isV) {
        const found = this._.value.find(v => isV ? this.#isValue(v) : this.#isCause(v));
        if (found === undefined) {
            // ここに new を追加！
            throw new JudgementError(`${isV ? '値' : 'Errorインスタンス'}が与えられていません。`);
        }
        return found;
    }

    #isValue(v) { return !this.#isCause(v) }
    #isCause(v) { return v instanceof Error }

    unwrap() { return this.#isCause(this.#first) ? this.throw() : this.at() }
    throw() { throw new JudgementError(`失敗。`, this.cause) }
    at(v) { return (v !== undefined) ? v : this.value }
}
```

### まとめ
エラーの主因は `new JudgementError` の `new` 忘れです。
そこを直せば、**「実装者が複数の失敗パターンを提示し、利用者が好きな方法で受け取る」** という、非常に柔軟で「お節介すぎない」理想的なAPIが完成します！
